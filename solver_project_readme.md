# Cube Solver Project Roadmap

이 문서는 큐브 타이머 프로젝트에 **실용적인 해 찾기 기능**과, 이후 **직접 구현한 탐색기 기반 solver**를 단계적으로 추가하기 위한 개발 계획서이다.

---

## 1. 프로젝트 목표

이 프로젝트의 최종 목표는 두 단계로 나뉜다.

### 1차 목표
기존 CubeTimer에 **실용적인 해(solution)** 를 빠르게 제시하는 기능을 추가한다.

이 단계에서는 다음을 만족하는 것을 목표로 한다.

- 현재 스크램블에 대해 해를 구할 수 있어야 한다.
- 브라우저에서 동작해야 한다.
- UI가 멈추지 않아야 한다.
- 결과를 텍스트와 시각화 형태로 보여줄 수 있어야 한다.

### 2차 목표
큐브 해를 찾는 과정을 외부 라이브러리에 전적으로 의존하지 않고, **직접 구현한 탐색기**로 대체하거나 병행한다.

이 단계에서는 다음을 목표로 한다.

- 큐브 상태 표현을 직접 설계한다.
- move table, pruning table, heuristic을 직접 구현한다.
- IDA* 기반 탐색기를 만든다.
- 장기적으로는 optimal solution 탐색까지 확장 가능하도록 구조를 잡는다.

---

## 2. 개발 방향

프로젝트는 다음 순서로 진행한다.

1. **기존 타이머 앱에 실용 solver 기능을 붙인다.**
2. **탐색기 구현에 필요한 내부 구조를 별도 모듈로 만든다.**
3. **간단한 퍼즐 또는 축소 상태공간으로 탐색기를 먼저 검증한다.**
4. **3x3용 탐색기로 확장한다.**
5. **필요하면 Rust/WASM 기반 고성능 코어로 이전한다.**

즉, 처음부터 완전한 optimal solver를 한 번에 만드는 것이 아니라,

- 먼저 "쓸 수 있는 기능"을 만들고
- 그다음 "직접 구현한 탐색기"를 붙이고
- 마지막에 "고성능화" 하는 방식으로 진행한다.

---

## 3. 우선순위

### 높은 우선순위

#### 3.1 실용 solver UI 추가
다음 기능을 먼저 구현한다.

- `해 찾기` 버튼
- 현재 스크램블 기준 solution 계산
- 결과 텍스트 표시
- 결과 복사 버튼
- solution move count 표시
- 가능하면 3D replay 표시

이 단계는 사용자가 즉시 체감할 수 있는 기능이기 때문에 가장 먼저 구현한다.

#### 3.2 solver 연산 분리
탐색 또는 solution 계산은 메인 UI를 멈추지 않도록 분리한다.

- Web Worker 사용
- 비동기 상태 관리
- 계산 중 UI 표시
- 취소/재시도 흐름 고려

#### 3.3 내부 상태 표현 설계
이후 직접 탐색기를 구현하기 위해 퍼즐 상태를 내부적으로 표현하는 구조를 잡는다.

예시:

- corner permutation
- corner orientation
- edge permutation
- edge orientation

이 단계부터는 단순 UI 기능이 아니라 solver 엔진의 기반 작업이 된다.

---

## 4. 단계별 계획

## Phase 1. 현재 앱에 실용 solver 추가

### 목표
현재 CubeTimer에서 선택된 스크램블에 대해 현실적인 해를 보여주는 기능을 제공한다.

### 할 일

- solve 버튼 UI 추가
- 결과 출력 영역 추가
- 스크램블 → 상태 변환 흐름 정리
- solution 결과를 화면에 표시
- move count 계산
- replay UI 연결

### 완료 기준

- 사용자가 버튼을 누르면 해가 나온다.
- 해를 복사할 수 있다.
- 해를 다시 볼 수 있다.
- 앱 사용성이 크게 떨어지지 않는다.

---

## Phase 2. 탐색기 실험용 모듈 분리

### 목표
직접 구현하는 solver 코어를 기존 앱 코드와 분리된 구조로 만든다.

### 권장 디렉터리 구조

```text
src/
  main.js
  solver/
    state.js
    moves.js
    search.js
    heuristic.js
    tables.js
    metrics.js
```

### 각 파일 역할

- `state.js`: 큐브 상태 표현
- `moves.js`: move 적용 함수
- `search.js`: IDA* 등 탐색 로직
- `heuristic.js`: lower bound 계산
- `tables.js`: pruning/move table 생성 및 로드
- `metrics.js`: move count, 통계 등 보조 기능

### 완료 기준

- UI 코드와 solver 코드가 분리된다.
- 탐색 코어를 독립적으로 테스트할 수 있다.

---

## Phase 3. 직접 구현한 탐색기 프로토타입

### 목표
간단한 상태공간에서 IDA* 기반 탐색기를 직접 구현해 본다.

### 추천 순서

1. 2x2 또는 축소 문제에서 시작
2. 상태 표현 검증
3. move 적용 검증
4. goal 판정 검증
5. heuristic 추가
6. IDA* 적용

### 핵심 포인트

- 탐색의 정답성
- heuristic이 admissible인지 확인
- pruning이 과도하게 정답을 제거하지 않는지 검증

### 완료 기준

- 작은 문제에서 최단해를 찾을 수 있다.
- 탐색 과정과 결과를 재현 가능하다.

---

## Phase 4. 3x3 solver 확장

### 목표
프로토타입을 실제 3x3 상태공간으로 확장한다.

### 구현 항목

- cubie representation 도입
- move table 생성
- orientation/slice 기반 pruning table 생성
- IDA* 깊이 제한 탐색 구현
- 중복 move pruning
- solution reconstruction

### 고려 사항

- 메모리 사용량
- 테이블 생성 시간
- 브라우저에서의 초기 로딩 비용
- 직렬화/캐싱 여부

### 완료 기준

- 제한된 범위라도 3x3 상태를 탐색 가능하다.
- 간단한 scramble에서 직접 구현한 탐색기로 해를 찾는다.

---

## Phase 5. 고성능화

### 목표
탐색 속도와 확장성을 개선한다.

### 선택지

#### 선택지 A: JavaScript 유지

- Web Worker 최적화
- TypedArray 사용
- move table 압축
- pruning table 압축
- 캐시 전략 개선

#### 선택지 B: Rust/WASM 이전

- solver 코어만 Rust로 분리
- WASM으로 브라우저 연결
- JS는 UI 담당, Rust는 탐색 담당

### 완료 기준

- 탐색 속도가 충분히 개선된다.
- 장기적으로 유지 가능한 구조가 된다.

---

## 5. 구현 원칙

### 5.1 한 번에 모든 것을 하지 않는다
처음부터 optimal 3x3 solver 완성본을 목표로 하지 않는다.

항상 다음 순서를 지킨다.

1. 작동하는 최소 버전 구현
2. 정확성 검증
3. 성능 개선
4. 기능 확장

### 5.2 UI와 solver 코어를 분리한다
타이머, 기록, 설정 UI와 탐색 코어를 섞지 않는다.

이유:

- 디버깅이 쉬워진다.
- 테스트가 쉬워진다.
- 나중에 Rust/WASM으로 교체하기 쉽다.

### 5.3 상태 표현을 먼저 고정한다
탐색기의 성능 문제를 해결하기 전에, 먼저 상태 표현과 move 적용 방식이 정확한지 검증한다.

### 5.4 측정 가능한 기준을 둔다
기능을 추가할 때마다 아래를 체크한다.

- 정답이 맞는가?
- 몇 초 걸리는가?
- 메모리를 얼마나 쓰는가?
- 브라우저에서 체감상 멈추는가?

---

## 6. 테스트 계획

### 단위 테스트

- move 적용 후 inverse move가 원상복구되는지
- solved state 판정이 정확한지
- heuristic이 0 이상인지
- heuristic이 admissible한지

### 통합 테스트

- scramble 적용 후 solution이 실제로 solved state를 만드는지
- solution 문자열이 정상 파싱되는지
- UI 표시 결과와 내부 상태가 일치하는지

### 성능 테스트

- 평균 탐색 시간
- 최대 탐색 시간
- 초기화 시간
- Web Worker 적용 전후 비교

---

## 7. 예상되는 어려움

### 7.1 상태공간 폭발
3x3는 상태공간이 매우 크므로 단순 탐색으로는 실용적인 성능이 나오지 않는다.

### 7.2 heuristic 설계
탐색 속도는 heuristic 품질에 크게 좌우된다.

### 7.3 브라우저 환경 제약
초기 테이블 생성, 메모리 사용, UI 멈춤 문제를 계속 관리해야 한다.

### 7.4 유지보수 복잡도
solver 코어가 커질수록 코드 가독성과 테스트 품질이 중요해진다.

## 10. 결론

이 프로젝트는 단순히 타이머 앱에 기능 하나를 추가하는 작업이 아니라,

- 큐브 상태를 모델링하고
- 탐색 알고리즘을 설계하고
- 성능과 정확성을 동시에 다루는

하나의 solver 프로젝트로 확장될 수 있다.

따라서 개발은 반드시 단계적으로 진행해야 하며,

- 먼저 쓸 수 있는 기능을 만들고
- 그다음 탐색기 구조를 직접 구현하고
- 마지막에 최적화하는 방식으로 간다.

이 문서는 그 로드맵의 기준 문서로 사용한다.
